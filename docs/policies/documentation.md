# ドキュメント配置ポリシー

## 目的

ドキュメントの一次情報を `docs/` に集約し、重複と陳腐化を防ぎます。
対象は、配置・導線・命名・更新手順です。

## 基本構造

- ルート `README.md`: プロダクト概要と入口リンク
- ルート `AGENTS.md`: エージェント運用の最小ルール
- `docs/`: 方針・手順・用語・テンプレートの一次情報ハブ

## ディレクトリ方針

- `docs/policies/`: 承認境界、開発原則、文書方針
- `docs/dev-practices/`: 実装・検証・反映の運用手順
- `docs/templates/`: 再利用テンプレート
- `docs/mcp/`: MCP 関連メモ
- `docs/glossary.md`: 用語集

新しいテーマを追加する場合は、曖昧な名前（`common`, `misc` など）を避け、目的が分かるディレクトリ名を使います。

## 運用原則

- 新しい一次情報は `docs/` に追加し、ルート文書は要約と導線のみ更新する
- 合意済み内容のみを正本に反映し、検討メモは `docs/scratch/` などの非正本領域で扱う
- 内容移動時は重複を残さず、旧位置には最小限の導線だけを残す
- Boy Scout Rule で、触れた範囲の古い記述を段階的に修正する

## 導線ルール

- 基本はトップダウン（上位 README → 下位 README → 個別文書）
- ただし、運用上の正本へ素早く到達するための直接リンク（例: `approval.md`）は許可する
- 相互参照を追加する際は、循環リンクや重複説明を増やさない
- リンク先が移動した場合は、同じ変更で参照元も更新する

## README 配置ルール

- `docs/` 配下の各ディレクトリには `README.md` を配置する
- 新しいディレクトリ追加時は、同コミットで `README.md` を追加する
- 検証は `tools/guards/no-docs.sh` を正本とする

## 命名ルール

- ディレクトリ・ファイル名は小文字ケバブケースを基本とする
- `README.md` は導線専用、詳細は個別文書へ分離する
- ドキュメントタイトルは内容が一意に判別できる名称を使う

## AGENTS.md 最小化

- AGENTS.md は「即時判断に必要な要点」に限定する
- 詳細手順、長文サンプル、背景説明は `docs/` 側へ移す
- AGENTS.md 更新時も `提案 → 明示承認 → 反映` を守る
