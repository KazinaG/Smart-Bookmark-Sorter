# 開発方針と品質原則

## 規約運用の目的と優先順位

開発規約は次の優先順位で運用します。

1. 事故防止
2. 最低限の変更追跡性
3. 認知負荷最小化
4. 意思決定速度

各項目の意味は次の通りです。

- 事故防止: 破壊的な操作や意図しない反映を防ぐ。
- 最低限の変更追跡性: 何を、なぜ、どこで変えたかを後から追える状態を保つ。
- 認知負荷最小化: 例外や重複を減らし、判断分岐を少なくする。
- 意思決定速度: 境界条件を固定し、承認の要否を短時間で判断できるようにする。

衝突時は上位の目的を優先します。速度のために例外を増やし、事故防止や追跡性を弱めることはしません。

## 基本原則

- 進め方は常に「提案 → 明示承認 → 反映」。
- 仕様と運用の一次情報は `docs/` に集約し、会話履歴を正本にしない。
- 不確実な事項は推測で進めず、調査または確認を先に行う。
- 重要な前提ずれや到達不能が発生した場合は、即座に共有して方針を再合意する。

## ハーネス委譲原則

リポジトリ内ドキュメントは、プロジェクト固有ルールだけを管理します。以下はハーネス側で担保されるため、重複記載しません。

- エージェントの実行環境・権限制御・エスカレーション手続き。
- 応答フォーマットやツール利用の一般ルール。
- 汎用的な安全ガード（例: 破壊的操作に対するデフォルト警戒）。

## 運用改善

- 運用中に乖離や摩擦が出た場合は、目的と優先順位に照らして改善案を作成する。
- 改善は小さく適用し、関連ドキュメントの導線と整合を同時に更新する。
- AGENTS.md を含む規約更新も「提案 → 明示承認 → 反映」を守る。
